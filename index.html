Styling maps nowadays is apart from the obvious design decisions not a big issue anymore thanks to the capabilities for MapServer or Mapnik which are widely used by various OSM projects. However, when preprocessing certain map elements such as hillshades or contour lines on a global scale, we couldn't find a framework which is as simple to use as for example TileMill for styling.

GDAL

  • gdaldem hillshade srtm.vrt hillshade.tif
  • gdal_contour srtm.vrt contours.shp

drawbacks

  1. not viable
  2. no customization
Yes, GDAL can both generate hillshading and contour lines out of a DEM via command line but (1) applying one of these commands to the global SRTM dataset (XXX x XXX) is not really viable and (2) there are no possibilities to refine the output by using image filters and so on.

Tile Pyramids

source: wikipedia

The main challenge here is that the source datasets are too big to be processed at once. Of course, this issue is solved in the design step by simply cutting the world into tiles and render the map in more easily digestible chunks.

tiled data --> CartoCSS/mapfile --> map tile

tiled data --> process --> hillshade tile

The nice thing about the styling step is that there is a style file (CartoCSS, Mapfile, ...) which is applied to the data chunks when rendering. Every WMS layer works that way or any tile cache was rendered this way. So, why not use this approach to preprocessing as well?

Mapchete

  • Python (scipy, numpy, shapely, ...)
  • chops data into tiles
  • applies user-defined process per tile
  • rasterio & Fiona for reading
  • spherical Mercator or geodetic (WGS84) tile pyramids

Command line utilities

  1. mapchete_serve your_process.mapchete
  2. mapchete_execute your_process.mapchete
There are two main command line utilities: mapchete_serve your_process.mapchete This command starts a web server hosting an OpenLayers instance where you can inspect your process outputs as you surf through the map.

mapchete_serve

  • starts web server (flask)
  • localhost:5000 --> OpenLayers
  • processes tiles on demand
  • always return freshly generated tiles: --overwrite
This is particularly useful if your source data set is too big to have it comfortably on your laptop. Ideally, you would imagine the development cycle like this: (1) write (2) run (3) (debug) (4) view However, when dealing with larger amounts of data, this is not realistic. If you have the data together with your process script on the server, you would have to download the result after every single run to view it on your local machine. You could also extract subsets of certain regions manually and develop your process using these regional datasets on your local machine, but then you would test your output just on these test regions.

Command line utilities

  1. mapchete_serve your_process.mapchete
  2. mapchete_execute your_process.mapchete
The second command line utility lets you "seed" your process into a full tile pyramid.

mapchete_execute

  • processes full pyramid
  • ... or just certain zoom levels: --zoom 0 8
  • ... or a specific tile (zoom, row, column): --tile 5 6 7
  • ... or a specific region: --bounds 10 40 12 50

Configuration

your_process.mapchete


%YAML 1.2
---
process_file: # path to process (python file)
input_files:
    file1: # path to file 1
    file2: # path to file 2
output:
    path: # absolute output path
    format: # PNG, PNG_hillshade, GTiff, GeoJSON, PostGIS or NumPy
    type: # geodetic or mercator
    dtype: # bool, uint8, uint16, int16, uint32, int32, float32, float64
    bands: # number of output bands
    nodata: # nodata value
The .mapchete file has a simple YAML syntax where you define the location of the Python file, the input files required for the process and the desired output format and location.

Read data

your_process.py


with self.open(
    self.params["input_files"]["raster_file"],
    resampling="bilinear"
    ) as my_raster_rgb_file:
    if my_raster_rgb_file.is_empty():
        return "empty"
    r, g, b = my_raster_rgb_file.read()
the built-in read functionality automatically reprojects and resamples the data to the tile bounds and resolution and returns the data as a masked numpy array: It also works for vector data. Features will be reprojected and clipped to the tile boundaries.
